MiU Request Lifecycle

Bir HTTP istegi geldikten sonra MiU icinde hangi katmanlarin calistigini, hangi kontrollerin devreye girdigini ve yanitin nasil olustugunu gosteren ozet sema. Bu akista hem statik route, hem dinamik router fallback, hem de guvenlik kontrolleri birlikte gosterilir.

Gorsel Akis

Kontrol Matrisi

Bakim Modu

Application::handle() icinde ilk kontrol budur. Aktifse router'a hic girilmez. Web tarafinda 503 HTML, API tarafinda 503 JSON doner.

API Rate Limit

Router::dispatch() basinda sadece /api/* path'lerinde calisir. Token varsa token bazli, yoksa IP bazli sinirlama yapar.

API Auth

ensureApiAuth() protected API endpoint'lerde token dogrular. Public action ise atlanir. Basariliysa auth bilgisi request attribute olarak yazilir.

Authorization

Controller::actionPermissions() tanimliysa router bunu merkezi olarak kontrol eder. Rol ve izinler DB'deki userRole.auth alanindan gelir.

CSRF

Sadece API disindaki POST isteklerde calisir. Token hataliysa request geldigi sayfaya veya ana dizine yonlendirilir.

Request Logger

Response olustuktan sonra devreye girer. Hassas alanlari maskeler; istek, response ve hata bilgisini log tablosuna yazar.

Notlar

Statik route once gelir

config/routes.php icindeki kayitlar ilk tercih edilir. make:site-page, make:admin-page ve make:api-endpoint komutlari bu dosyaya otomatik kayit yazar.

Dinamik router fallback olarak kalir

Statik route bulunamazsa controller ve action isimleri path'ten cozulur. Basarili cozumler cache'e yazilir; ama auth, permission ve csrf kontrolleri yine her istekte yeniden calisir.

Controller ince kalacak sekilde tasarlandi

Controller request alir, gerekli servisi cagirir, response dondurur. Is kurallari servislerde, veri erisimi ise model katmaninda toplanir.
